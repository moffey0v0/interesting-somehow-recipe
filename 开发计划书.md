# 开发计划书

要将“冰箱炼金术”从一个想法落地为可以直接在网上访问的完整交互式Web应用，需要经历从 UI 设计、前端开发、AI (Prompt) 调试到最终部署的完整流程。以下是详细的阶段性开发计划。

## 阶段一：原型设计与资产准备 (规划期)
**目标：确定视觉规范，找到合适的素材，跑通核心逻辑。**
1.  **架构梳理与技术选型**：
    *   **前端框架**：推荐使用 **React (结合 Next.js)** 或 **Vue (结合 Vite)**。因为应用涉及大量的状态管理（已选食材数组、模式开关、口味滑块等）。
    *   **样式/动画**：使用 TailwindCSS 快速布局，结合 Framer Motion (React) 或 GSAP 处理复杂的烹饪动画、食材飞入特效。
    *   **后端/AI交互**：采用 Serverless 架构（Next.js API Routes 或独立 Node.js/Python 后端），对接大语言模型 API（如 OpenAI GPT-4o, Anthropic Claude 3.5 Sonnet, 或 DeepSeek 以节约成本）。
2.  **UI/UX 线框图与视觉设计**：
    *   绘制左栏风格设定区、中部烹饪锅操作区、右栏食材选取区的基本布局。
    *   确定“现代整洁厨房 + 微量魔法点缀”的色彩基调（如木质色调 + 发光按钮等）。
3.  **美术资产收集/生成**：
    *   寻找或使用 AI 生图工具（Midjourney/DALL-E）生成各种食材的卡通图标（要求背景透明的 PNG 或 SVG）。
    *   准备“锅”、“厨房台面”、“小纸条底图”、“成功/失败动画帧或GIF”素材。

## 阶段二：前端界面与交互开发 (基础建设期)
**目标：构建一个看得见、摸得着，可以点来点去的静态网页。**
1.  **基础页面搭建**：
    *   切分布局（左、中、右栏）。实现中英文语言切换的 i18n 国际化包。
2.  **组件开发与状态绑定**：
    *   **左区（风格设定）**：开发菜系两级选择器、Slider（滑动条）和 Toggle（开关），将它们的值绑定至全局状态管理库（如 Zustand）。
    *   **中区（烹饪锅）**：实现食材槽的陈列与删除逻辑，以及烹饪触发。
    *   **右区（食材选取）**：实现分类 Tab 切换，渲染食材列表；实现“自定义食材”发光菜篮按钮及输入流。
3.  **核心动画实现**：
    *   实现点击食材时，食材缩小并以抛物线飞进“锅”里的动画轨迹。
    *   当点击“锅”时，根据当前全局状态，触发“煮饭”动画或“爆炸着火”动画。

## 阶段三：AI 核心大脑开发与 Prompt 工程 (灵魂注入期)
**目标：让 AI 听懂用户的复杂要求，并返回格式化、极其稳定且安全的数据。**
1.  **安全词库/规则前置拦截**：
    *   在前端或后端维护一个轻量级的“危险搭配词库”或“不可生食名单”（如河豚、生豆角、奇怪化学品等）。
    *   在请求 AI 之前，先进行一次代码层面的校验，如果命中危险组合，直接在前端播放“爆炸失败”动画，阻止请求。
2.  **Prompt 架构设计**：
    需要精心设计 System Prompt，使其能同时理解：选定的食材列表及数量、菜系、生存/创造模式、五维口味分布、狂野程度。
    *   *Prompt 示例结构*：
        ```text
        你是一个极具创意的“疯狂厨师”。你现在面临以下挑战：
        模式：【生存模式】（不可使用未提供的主辅料，只能用常规调料）
        提供食材：西红柿x2，老干妈x1，牛奶x1，面条x1
        要求口味：极辣，偏甜。狂野程度：90/100。
        请严守食品安全底线（绝对不能吃死人或中毒）。
        请输出 JSON 格式（必须严格遵守），包含：菜名、翻车率预估、消耗食材列表、新增食材列表（生存模式下必须为空）、详细步骤、魔改来源。结果必须是做法非常明确的菜谱，不要出现任何无意义的修饰词或者比喻形容。
        ```
3.  **API 联调与 JSON 解析**：
    *   构建后端转发服务，保护你的 API Key 不被泄露。
    *   将 AI 返回的 JSON 串解析，并转换为前端“小纸条”所需的数据结构。

## 阶段四：特色功能开发 (完善期)
**目标：实现图片生成导出功能与细节打磨。**
1.  **图片生成机制 (Canvas/HTML-to-Image)**：
    *   引入 `html2canvas` 或类似库。
    *   当用户点击“小纸条”查看详情时，弹出一个精美的 HTML DOM 模版，渲染菜名、步骤和来源。
    *   允许用户点击“保存为图片”，通过代码将该 DOM 转换为海报图片下载。
2.  **极端输入边界测试**：
    *   测试全输入“水”、全输入“黄油”、或者乱输入非食材（如：皮鞋）时的 AI 表现，优化 Prompt 使其能以幽默方式回绝或强行合理化（“熬煮皮鞋糖浆”）。

## 阶段五：测试、部署与发布上线
**目标：让全世界都能玩到这个网站。**
1.  **全面测试**：
    *   手机端适配检查：由于移动端屏幕小，可能需要将左中右三栏布局改为“底部抽屉（控制台）+ 上方锅 + 滑动食材区”的形式。
2.  **域名与托管部署**：
    *   **前端部署**：将应用打包，推送至 GitHub，使用 Vercel 或 Netlify 进行免费托管并配置自动构建。
    *   **域名配置**：购买一个有趣的域名（如 `interesting-somehow-recipe.com` ），解析到 Vercel。
3.  **上线后运营**：
    *   接入 Google Analytics 等分析工具，看看用户最喜欢用什么奇怪的食材。
    *   设立一个“反馈/捐赠”按钮，收集用户做出来的神奇菜品截图。
